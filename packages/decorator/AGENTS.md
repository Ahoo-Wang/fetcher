## Project Structure

```
src/
  apiDecorator.ts             — @api() class decorator (basePath, options)
  endpointDecorator.ts        — @get/@post/@put/@delete/@patch method decorators
  parameterDecorator.ts       — @path/@query/@header/@body parameter decorators
  requestExecutor.ts          — Auto-generated method execution via reflect-metadata
  reflection.ts               — Reflect-metadata utility functions
  functionMetadata.ts         — Function metadata definitions
  generated.ts                — autoGeneratedError() placeholder for decorated methods
  endpointReturnTypeCapable.ts — Return type handling for endpoints
  executeLifeCycle.ts         — Lifecycle hooks for request execution
  index.ts                    — Barrel export
  stories/                    — Storybook stories
```

### Key Concepts

- **@api(basePath, options)**: Class-level decorator that defines the service base path and Fetcher configuration
- **@get/@post/@put/@delete/@patch**: Method decorators that define HTTP verb and path
- **@path/@query/@header/@body**: Parameter decorators that bind method arguments to request parts
- **reflect-metadata**: Used to store and retrieve decorator metadata at runtime
- **Auto-implementation**: Decorated methods throw `autoGeneratedError()` as placeholder; the decorator system replaces them with actual fetch calls
- **RequestExecutor**: Resolves metadata into actual Fetcher calls

## Boundaries

- ✅ Adding new parameter decorators (e.g., @header)
- ✅ Adding new lifecycle hooks
- ✅ Writing new tests
- ⚠️ Adding new method decorators — must integrate with requestExecutor
- ⚠️ Changing reflect-metadata key formats — affects generated code
- 🚫 Changing @api() or endpoint decorator signatures (breaking change)
- 🚫 Removing autoGeneratedError() pattern without migration plan
- 🚫 Breaking compatibility with generated code from fetcher-generator
